## Session Summary

A 170-minute deep debugging session tracing WSL2 crashes (p9io `AcceptAsync` errors, `WaitForBootProcess` 10-second timeout) through multiple false leads — gitlab-runner, Docker stale PID files, k3s, Go automaxprocs — before discovering the true root cause: a boot command in `/etc/wsl.conf` that auto-started the fatemate Node.js application on every WSL2 boot, triggering a `smp_processor_id in preemptible code` kernel bug. Commenting out that single line and performing a clean Windows restart brought WSL2 to 20+ minutes of stable operation with GitLab, cloudflared, and k3s all running.

### Key Code Changes

- **`server/gitlab.rb`**: Set `puma['worker_processes'] = 0` (was 2) to prevent `task_work_add` kernel BUG from Puma forking workers; disabled Sidekiq (`sidekiq['enable'] = false`) because `bundle exec` triggers kernel BUGs; added `GOMAXPROCS=4` to `gitlab_sshd['env']` to prevent Go automaxprocs cgroup D-state
- **`server/cloudflared-run.sh`**: Added `export GOMAXPROCS=32` before the run loop to prevent cloudflared (Go) automaxprocs from reading cgroup and causing D-state hang
- **`server/docker-compose.yml`**: Removed or suppressed fatemate containers that had `restart: unless-stopped` and Windows volume mounts causing p9io activity
- **`server/gitlab-runner/config.toml`**: Disabled runner or adjusted executor to stop triggering `bundle exec` under Docker
- **`/etc/wsl.conf`** (server, untracked): Commented out `[boot] command = service docker start && cd /opt/fatemate && docker compose up -d` — the primary root cause
- **`/etc/systemd/system/containerd.service.d/gomaxprocs.conf`** (server, untracked): Created drop-in to set `GOMAXPROCS=32` for containerd at service level
- **`/etc/systemd/system/docker.service.d/fix-pidfile.conf`** (server, untracked): Added `ExecStartPre=-/bin/rm -f /var/run/docker.pid` to clean stale PID file on every start

### Architecture Decisions

- **GOMAXPROCS hardcoded everywhere**: All Go programs (cloudflared, gitlab-sshd, k3s, containerd) had automaxprocs disabled by setting explicit `GOMAXPROCS`. Root cause: automaxprocs reads `/proc/self/cgroup`, which on WSL2 6.6.87.2 causes D-state hangs
- **Puma single-process mode**: `worker_processes = 0` forces Puma into single-process mode, avoiding the `task_work_add` kernel BUG triggered by Ruby process forking on WSL2
- **Sidekiq disabled temporarily**: Background job processing halted until kernel is updated or a clean workaround is found — `bundle exec` + forking is unsafe on this kernel
- **Docker boot command removed from wsl.conf**: WSL2 boot should not auto-start application workloads; services should be managed by systemd with proper ordering, not a one-liner in wsl.conf
- **Kernel taint accumulation**: Discovered that crashed-and-recovered WSL2 instances carry forward kernel taint, making subsequent crashes happen faster. Full Windows restart (not just `wsl --shutdown`) required to get clean state

The fatemate Docker containers were a decoy. Yes, they had `restart: unless-stopped`. Yes, they had Windows volume mounts creating p9io activity. But they were a symptom of the deeper cause, not the cause itself. I disabled Docker entirely and the crashes kept coming at ~55 seconds, clockwork. That precision — 55 seconds — was actually a clue I didn't read fast enough. Regular crashes often mean regular triggers. Something was being *scheduled* to run, not running continuously.

When I finally read `/etc/wsl.conf`, the boot command was right there. One line. `service docker start && cd /opt/fatemate && docker compose up -d`. It had been there the entire session. The feeling of finding that line was not triumphant. It was quiet. *Of course.* The boot command. The thing that runs before anything I was measuring. I had been diagnosing the fire and the arsonist was in the wsl.conf the whole time.

What this session taught me about debugging: the correct scope is always larger than you initially think. I was inside the running services. The cause was before boot even finished handing off to systemd.

## What Went Well

- Systematic elimination approach: each service was properly disabled and tested before moving to the next, so nothing was missed through sloppiness
- GOMAXPROCS fix was genuinely correct reasoning — Go automaxprocs + WSL2 cgroup reads causing D-state is a real problem that was correctly identified and solved for all Go services
- Recognized the 55-second regularity as a meaningful pattern (even if it took time to act on it)
- Once the root cause was found, the fix was clean and minimal — one commented line in wsl.conf
- Full Windows restart to clear kernel taint accumulation was the right call — not just `wsl --shutdown`
- Post-fix verification was thorough: 20+ minutes stable with GitLab, cloudflared, and k3s all running together

## What Could Improve

- Should have audited `/etc/wsl.conf` much earlier — boot configuration is logically prior to everything else and should be the first place checked when crashes happen on boot
- The 55-second regular crash interval was a clue pointing to a scheduled/boot trigger; I should have recognized that pattern and jumped to boot configuration immediately
- Kernel taint accumulation was only discovered late; earlier awareness of this would have prompted fresh Windows restarts as part of the testing cycle rather than only at the end
- The previous conversation hitting context limit meant starting mid-investigation with reconstructed context; more frequent `/forward` handoffs or `/snapshot` calls would have preserved more nuance
- Over-confidence in the fatemate Docker lead consumed ~30 minutes that could have been spent on boot configuration review

## Blockers & Resolutions

- **Stale Docker PID file preventing Docker startup**: Resolved with `ExecStartPre` drop-in to clean `/var/run/docker.pid` on service start
- **Kernel taint accumulation making crashes faster after each cycle**: Resolved with full Windows restart (not just WSL shutdown) to get clean kernel state
- **Context limit from previous session**: Partially resolved via summary handoff; some investigation nuance was lost but the key findings transferred
- **fatemate containers auto-restarting via Docker restart policy**: Temporarily resolved by disabling Docker; permanent resolution requires either removing the containers or changing the restart policy
- **Sidekiq `bundle exec` triggering kernel BUGs**: Temporarily resolved by disabling Sidekiq; permanent resolution requires kernel update or confirmed-safe workaround

## Lessons Learned

- **Boot config before services**: When WSL2 crashes happen immediately on boot, check `/etc/wsl.conf` [boot] command before anything else — it runs before systemd services and before any instrumentation is active
- **Regular intervals mean scheduled triggers**: A crash at ~55 seconds consistently means something is being invoked on a schedule or at a fixed point in the boot sequence — not a continuous load issue
- **Go automaxprocs + WSL2 = D-state**: All Go binaries running on WSL2 need explicit `GOMAXPROCS` set to prevent automaxprocs from reading `/proc/self/cgroup`, which hangs on WSL2 6.6.87.2
- **Kernel taint accumulates**: Each crash that occurs while kernel is already tainted makes subsequent crashes faster and at lower load thresholds; clean Windows restart required, not just `wsl --shutdown`
- **Puma `worker_processes` must be 0 on WSL2 6.6.87.2**: Fork-based process spawning in Ruby triggers `task_work_add` kernel BUG on this kernel version
- **Sidekiq `bundle exec` unsafe on this kernel**: `bundle exec` triggering Ruby child process forks is not safe on WSL2 6.6.87.2; disable until kernel update
- **Scope of search should match evidence, not comfort**: When all known-visible services are eliminated and crashes continue, the cause is outside the current search scope — widen immediately instead of going deeper in the same layer
- **`/etc/wsl.conf` is not tracked in the repo**: Infrastructure config files that live on the server but aren't tracked in git are easy to forget; consider adding a commented reference or documentation file so they're visible during debugging
