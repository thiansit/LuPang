### Friction Points (3 required)

1. **Silent subshell failure with `set -eo pipefail`**: When a background subshell exits before printing any output, CI traces show nothing — no error, no context. This makes root cause analysis rely entirely on reproducing the environment exactly. Suggestion: Always add a `trap 'echo FAILED at line $LINENO' ERR` in subshells, or at minimum `set -x` during debugging.

2. **Stale manual test artifacts polluting CI**: Running `docker build` as `benz` directly creates files that `gitlab-runner` can't overwrite. This is a class of bug that only appears when the same machine hosts both manual developer work and CI. Suggestion: Make CI always use ephemeral `/tmp/build-$CI_JOB_ID-$svc.log` names, never fixed names that can collide.

3. **Bad `sed` edit went undetected until pipeline retry**: I modified line 261 with `sed`, produced backwards logic, and pushed it without verifying. The pipeline then failed for a different reason before reaching that line. Suggestion: After any shell-based file edit, immediately `grep` the changed line to confirm it looks right — never trust regex-based edits without verification.

## Lessons Learned

- **Pattern**: `VAR=$(cmd | pipeline) || VAR=""` is the correct pattern for any command substitution that might fail in a `set -eo pipefail` shell. The `|| VAR=""` catches the non-zero exit and prevents script termination. Without it, even `grep -v` on empty stdin silently kills the subshell.

- **Mistake**: Used `sudo -u gitlab-runner` without `-H` → HOME pointed to the wrong directory, making test results unreliable. Always use `sudo -Hu username` when simulating a different user's shell environment.

- **Discovery**: The `ApplicationSet` CRD can be missing from ArgoCD if the original install was incomplete or if k3s state was partially restored. The CRD is too large for client-side apply (>262KB annotation) — must use `kubectl apply --server-side`.

- **Pattern**: When WSL2 restarts, kernel modules in `/etc/modules-load.d/*.conf` must be loaded before k3s starts. The `xt_nat` module is required for iptables NAT rules that kube-proxy writes. CoreDNS depends on kube-proxy having populated `KUBE-SERVICES` before it can resolve cluster DNS.

- **Mistake**: Verified fixes in test scripts using `heredoc` but production CI uses `bash -c` with exported variables. The subtle difference in how subshells inherit `set -eo pipefail` caused false confidence in test results.
