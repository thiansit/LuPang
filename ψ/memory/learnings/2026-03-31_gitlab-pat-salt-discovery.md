# GitLab 18.x PAT Token Digest Uses Salted SHA256

**Date**: 2026-03-31
**Context**: WSL2 + GitLab 18.10.1 infrastructure session — manual PAT token insertion
**Confidence**: High (verified by reading source code + successful auth after applying)

## Key Learning

GitLab 18.x does **not** use plain SHA256 for Personal Access Token digest verification. It uses a salted SHA256 where the salt is derived from the first 32 characters of `db_key_base` in `/etc/gitlab/gitlab-secrets.json`.

This is implemented in `/opt/gitlab/embedded/service/gitlab-rails/lib/gitlab/crypto_helper.rb`:

```ruby
def sha256(value)
  salt = Gitlab::Encryption::KeyProvider[:db_key_base_truncated].encryption_key.secret
  ::Digest::SHA256.base64digest("#{value}#{salt}")
end
```

The `db_key_base_truncated` key provider extracts exactly the first 32 characters of `db_key_base`. The digest stored in `personal_access_tokens.token_digest` column is `base64(SHA256(token_plaintext + salt))`.

**When manually inserting a PAT token into PostgreSQL**, you must compute the digest this way:

```bash
TOKEN="glpat-your-token-value"
SALT=$(sudo python3 -c "
import json
with open('/etc/gitlab/gitlab-secrets.json') as f:
    d = json.load(f)
print(d['gitlab_rails']['db_key_base'][:32])
")
DIGEST=$(printf '%s' "${TOKEN}${SALT}" | openssl dgst -sha256 -binary | base64 -w0)

sudo -u gitlab-psql /opt/gitlab/embedded/bin/psql -h /var/opt/gitlab/postgresql -d gitlabhq_production -c \
  "UPDATE personal_access_tokens SET token_digest='${DIGEST}' WHERE name='your-token-name';"
```

## The Pattern

```
digest_stored_in_db = base64( SHA256( token_plaintext + salt ) )
salt                = db_key_base[0..31]   # first 32 chars only
db_key_base         = from /etc/gitlab/gitlab-secrets.json → gitlab_rails.db_key_base
```

This applies to:
- Personal Access Tokens (`personal_access_tokens` table, `token_digest` column)
- Runner registration tokens use a similar but separate mechanism
- Group/project access tokens follow the same PAT pattern

## Why This Matters

When GitLab's HTTP interface is unreliable (e.g., WSL2 p9io instability causing puma restarts), direct database token manipulation becomes the emergency auth path. Without knowing the salted digest formula, every manually-inserted token returns 401 despite being syntactically valid.

This formula is not prominently documented in GitLab's public docs — it requires reading `crypto_helper.rb` and tracing the `KeyProvider[:db_key_base_truncated]` implementation. The discovery took ~45 minutes of source code archaeology.

**Secondary path**: Use GitLab admin API impersonation tokens instead of manual PAT insertion. Impersonation tokens are generated by GitLab itself with the correct format and digest, avoiding the manual computation entirely:

```bash
# Create impersonation token via admin API
curl -s --header "PRIVATE-TOKEN: ${ADMIN_PAT}" \
  "http://localhost:8181/api/v4/users/${USER_ID}/impersonation_tokens" \
  --data "name=emergency-token&scopes[]=api&scopes[]=read_repository&scopes[]=write_repository"
```
